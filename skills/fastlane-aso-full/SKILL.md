# Fastlane ASO Full — 본격 스토어 최적화

## 개요

심사가 필요한 핵심 메타데이터를 포함한 전체 ASO 최적화:

**iOS (deliver)**

- 부제목 (30자) — 앱 이름 바로 아래 노출, 검색 가중치 높음
- 키워드 (100자, 쉼표 구분) — 검색 노출의 핵심
- 설명 (4,000자) — 전환율에 영향
- 출시 노트 — 업데이트마다 작성

**Android (supply)**

- 전체 설명 (4,000자) — 검색 + 전환 모두에 영향
- 출시 노트 — 업데이트마다 작성

**절대 변경하지 않는 항목: 앱 이름**

---

## 실행 흐름

### Step 1. 앱 정보 수집

사용자에게 아래 정보를 받는다:

- 앱스토어 또는 플레이스토어 URL (하나 이상)
- 타겟 로케일 목록 (기본: ko, en-US, ja)
- 이번 업데이트의 핵심 변경사항 (출시노트용, 선택)
- 특별히 강조하고 싶은 기능이나 방향 (선택)

### Step 2. 심층 앱 분석

웹 검색을 통해 철저하게 분석한다. 10억 다운로드를 위해 대충 하지 않는다.

#### 2-1. 자사 앱 분석

- 현재 스토어 메타데이터 전체 확인
- 앱의 핵심 기능, USP(고유 판매 제안), 타겟 유저 페르소나
- 현재 카테고리 및 서브카테고리 적합성 평가
- 유저 리뷰에서 자주 언급되는 키워드/불만/칭찬 추출

#### 2-2. 경쟁 앱 분석

- 같은 카테고리 상위 10개 앱의 메타데이터 조사
- 키워드 전략 역분석: 어떤 키워드로 상위 노출되고 있는지
- 설명 구조 패턴: 어떤 순서로 어떤 내용을 배치하는지
- 경쟁 앱이 놓치고 있는 키워드 기회 발굴

#### 2-3. 키워드 리서치

- 해당 카테고리의 고검색량 키워드 목록 작성
- 경쟁 강도 대비 검색량이 높은 롱테일 키워드 발굴
- 시즌성 키워드 및 트렌드 키워드 파악
- 각 로케일별로 별도 키워드 리서치 수행 (번역 X, 현지 검색 트렌드 기반)

### Step 3. 메타데이터 생성

#### 3-1. iOS 키워드 (100자)

**생성 규칙:**

- 쉼표로 구분, 공백 넣지 않음 (글자 수 절약)
- 앱 이름/부제목에 이미 포함된 단어는 중복 사용하지 않음 (Apple이 자동 인덱싱)
- 단수형만 사용 (복수형은 Apple이 자동 매칭)
- 경쟁 앱 이름은 넣지 않음 (정책 위반 리스크)
- 검색량 높은 키워드를 앞쪽에 배치
- 로케일별로 완전히 다른 키워드 세트 작성 (번역 X)

각 로케일별로 2가지 버전 제시:

- 버전 A: 검색량 중심 (대중적 키워드 위주)
- 버전 B: 틈새 공략 (롱테일 + 경쟁 낮은 키워드 위주)

#### 3-2. iOS 부제목 (30자)

**생성 규칙:**

- 핵심 키워드를 자연스럽게 포함
- 앱 이름과 중복되는 단어 피함
- 앱의 핵심 가치를 한 줄로 전달
- 각 로케일별 네이티브 표현 사용

각 로케일별로 2가지 버전 제시.

#### 3-3. iOS 설명 (4,000자)

**생성 규칙:**

- 첫 3줄이 승부처 (펼치기 전에 보이는 영역)
- 첫 3줄에 핵심 가치 제안 + 강력한 훅
- 주요 기능은 이모지 + 볼드 활용하여 스캔하기 쉽게
- 소셜 프루프 (수상, 미디어 언급, 유저 수 등) 포함
- 핵심 키워드를 자연스럽게 본문에 녹임
- CTA로 마무리
- 각 로케일별 문화에 맞는 톤앤매너 적용

**구조 템플릿:**

```
[훅 — 핵심 가치 한 줄] (1-2줄)
[소셜 프루프 또는 수상 실적] (1줄)

[핵심 기능 1] (2-3줄)
[핵심 기능 2] (2-3줄)
[핵심 기능 3] (2-3줄)
...

[유저 후기 또는 미디어 인용] (선택)
[CTA — 지금 시작하세요] (1줄)
```

#### 3-4. Android 전체 설명 (4,000자)

**iOS 설명과 다른 점:**

- Google Play는 설명 본문의 키워드를 검색 인덱싱에 반영함 → 키워드 밀도가 중요
- 핵심 키워드를 설명 전체에 걸쳐 자연스럽게 5-7회 반복
- 단, 키워드 스터핑(부자연스러운 반복)은 패널티 대상
- HTML 태그 사용 불가, 이모지와 줄바꿈으로 가독성 확보

#### 3-5. 출시노트

**생성 규칙:**

- 핵심 변경사항을 간결하게
- 유저 피드백 반영했다는 점 강조 (유저가 소중하다는 느낌)
- 가벼운 톤 권장 (앱 브랜드에 맞게 조절)
- 각 로케일별 네이티브 표현

**사용자에게 모든 버전을 보여주고 선택/수정 요청을 받은 후 다음 단계로 진행한다.**

### Step 4. Fastlane 메타데이터 파일 생성

사용자가 확정한 문구를 Fastlane 메타데이터 구조에 맞게 파일로 생성한다.

#### iOS 디렉토리 구조

```
ios/fastlane/metadata/
├── ko/
│   ├── subtitle.txt
│   ├── keywords.txt
│   ├── description.txt
│   └── release_notes.txt
├── en-US/
│   ├── subtitle.txt
│   ├── keywords.txt
│   ├── description.txt
│   └── release_notes.txt
└── ja/
    ├── subtitle.txt
    ├── keywords.txt
    ├── description.txt
    └── release_notes.txt
```

#### Android 디렉토리 구조

```
android/fastlane/metadata/android/
├── ko-KR/
│   ├── full_description.txt
│   └── changelogs/[version_code].txt
├── en-US/
│   ├── full_description.txt
│   └── changelogs/[version_code].txt
└── ja-JP/
    ├── full_description.txt
    └── changelogs/[version_code].txt
```

**Android 출시노트 주의:**

- changelogs 폴더 아래에 `[version_code].txt` 형식으로 생성
- version_code는 사용자에게 확인받는다 (android/app/build.gradle의 versionCode)

**각 파일에는 확정된 문구만 들어가며, 파일 끝에 불필요한 줄바꿈을 넣지 않는다.**

### Step 5. 완료 안내

메타데이터 파일 생성까지만 수행한다. **배포(deliver/supply)나 심사 제출은 하지 않는다.**

사용자에게 아래를 안내한다:

- 생성된 파일 경로와 내용 요약
- iOS는 부제목·키워드·설명 변경 시 앱 심사가 필요하다는 점
- 파일 확인 후 배포는 사용자가 직접 판단하여 실행

## 주의사항

- iOS 부제목·키워드·설명은 **앱 심사가 필요**하다 (보통 24-48시간)
- 키워드 변경 후 검색 순위 반영까지 며칠 걸릴 수 있다
- 한 번에 모든 로케일을 바꾸기보다, 주요 로케일(ko, en-US)부터 실험 후 확장 권장
- 키워드에 경쟁 앱 이름이나 브랜드명을 넣지 않는다 (정책 위반)
- Android는 설명 본문이 검색 인덱싱에 반영되므로 키워드 배치가 매우 중요
- 앱 이름은 절대 변경하지 않는다
- 설명에 가격 정보, 시한성 프로모션 등 빠르게 변하는 정보는 넣지 않는다
